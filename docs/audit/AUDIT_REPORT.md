## Not consolidated (intentional or unsafe)

These exact-duplicate or near-duplicate groups were detected but **left
intact** because consolidation would either remove context or break a build
artifact. Each is recorded in [`MANUAL_REVIEW.md`](MANUAL_REVIEW.md).

* `examples/vectorvroom/vendor/ruvector/**` — vendored copies of crates;
  vendored copies must remain self-contained.
* `crates/*/pkg/README.md` — `wasm-pack` regenerates these on every build,
  so editing them would be wiped on the next `wasm-pack build`. Consider
  adding `crates/*/pkg/` to `.gitignore` instead.
* `npm/packages/<crate>/README.md` ↔ `crates/<crate>/README.md` — these are
  intentionally dual-published READMEs (one for crates.io, one for npm).
* `.claude/agents/<flat>` ↔ `.claude/agents/<nested>` — claude-flow
  references both flat and nested paths; consolidating risks breaking agent
  spawn lookups.

## Contradictions detected

The pipeline scans for numeric claims (`ruvector vX.Y.Z`, `rust X.Y`,
`node X`, `<N>x speedup`) and reports any documents that disagree on the
value of the same claim.

The full list of value clusters is in
[`MANUAL_REVIEW.md`](MANUAL_REVIEW.md). The most actionable items are:

1. **`ruvector` version drift** — `README.md` declares `0.3.0`, but
   `docs/research/sublinear-time-solver/00-executive-summary.md` and
   `docs/research/wasm-integration-2026/00-executive-summary.md` reference
   `2.0.3`, `ui/ruvocal/docs/adr/ADR-029-...md` references `2.0.1`, and
   older docker-integration reports reference `0.1.x`. Manual triage
   needed to decide which research docs are aspirational vs. shipped.
2. **ADR number collision** — `docs/adr/ADR-144-diskann-vamana-implementation.md`
   and `docs/adr/ADR-146-diskann-vamana-implementation.md` are byte-identical.
   One ADR number must win; flagged in `MANUAL_REVIEW.md`.
3. **Cross-tree ADR collision** — `docs/adr/ADR-026-rvcow-...md` is 99.7%
   identical to `crates/rvf/docs/adr/ADR-031-rvcow-...md`. The same content
   is recorded under two different ADR numbers in two different ADR
   registries.
4. **Per-crate Rust MSRV drift** — declared MSRVs span 1.70 → 1.91.1 across
   crates. This is partly intentional (crates with different feature sets)
   but worth aligning where possible.

## Methodology

The audit pipeline lives in
[`scripts/audit/detect_dups.py`](../../scripts/audit/detect_dups.py) and
[`scripts/audit/consolidate.py`](../../scripts/audit/consolidate.py). It is
deterministic and can be re-run in CI to catch new duplicates as the docs
grow.

* **Normalisation** — strips YAML front-matter, collapses whitespace, and
  lowercases content before hashing.
* **Exclusion** — `node_modules/`, `target/`, `.git/`, `.cargo/`, `dist/`,
  `build/`, `.next/`, `.turbo/`, `.pnpm-store/`, `.planning/` are skipped.
* **Hash bucket** — exact duplicates are collapsed via SHA-256 of the
  normalised content.
* **Near-dup bucket** — files are bucketed by minhash of the first sorted
  shingle and any pair with Jaccard ≥ 0.70 (over 7-word shingles) is
  reported. A coarse prefix bucket catches pairs the minhash bucket missed.
* **Contradictions** — regex-based extraction of numeric claims followed by
  a value-cluster cross-reference.

## How RuVector is used in the audit

The audit scripts intentionally do **not** require RuVector to detect
duplicates (so the pipeline can run in restricted CI environments). However,
the n8n integration in [`scripts/n8n/`](../../scripts/n8n/) wires the
knowledge base into RuVector for *semantic* retrieval, which complements
the pipeline's syntactic checks.

See [`scripts/n8n/README.md`](../../scripts/n8n/README.md) for the runtime
integration architecture.
